Part 1 · ผู้ใช้งานประจำวัน

เริ่มใช้งานอย่างเป็นลำดับ

คู่มือนี้อธิบายงานประจำวันของ ERP ตั้งแต่ขาย รับเข้า ควบคุมสต็อก จัดซื้อ จนถึงรับชำระเงิน เมนูที่เห็นและปุ่มที่กดได้ขึ้นอยู่กับสิทธิ์ที่องค์กรกำหนดไว้ จึงให้ทำรายการผ่านเมนูที่ได้รับสิทธิ์เท่านั้น

ก่อนกดยืนยัน: ตรวจคลังสินค้า รายการสินค้า จำนวน ราคา ลูกค้า/ผู้ขาย และวันที่เอกสารให้ครบ เพราะเอกสารที่ยืนยันแล้วอาจตัดหรือเพิ่มสต็อก และบางประเภทจะลงบัญชีตามการทำงานของระบบ

ข้อมูลที่ควรพร้อมก่อนเริ่ม

  • ลูกค้าและผู้ขายที่ใช้งาน
  • สินค้า หมวดหมู่ หน่วยนับ และรายการราคา
  • คลังสินค้าสำหรับรับ จ่าย หรือโอน
  • ข้อมูลภาษีและงวดภาษีที่เปิดอยู่สำหรับใบแจ้งหนี้

รูปแบบการทำงานร่วมกัน

  • บันทึกเป็น ร่าง เมื่อต้องตรวจทานหรือรออนุมัติ
  • กด ยืนยัน เมื่อข้อมูลและเอกสารต้นทางถูกต้อง
  • ดูเลขที่เอกสารและสถานะก่อนสร้างเอกสารถัดไป
  • ใช้หน้ารายการเพื่อค้นหา ตรวจยอด และติดตามสถานะ

ไปที่หัวข้อ

การขาย คลังสินค้า จัดซื้อ รับชำระเงิน LINE ผู้ดูแล

01 · การขาย

ใบเสนอราคา ถึงใบส่งสินค้าและใบแจ้งหนี้

เมนูหลัก: การขาย › ใบเสนอราคา, ใบสั่งขาย, ใบส่งสินค้า

ขั้นตอนหลัก สร้างและยืนยันใบสั่งขาย ▼
  1. 1. หากใช้ใบเสนอราคา ให้ตรวจราคาและรายการก่อนเปลี่ยนเป็นการขาย
  2. 2. สร้างใบสั่งขาย เลือกลูกค้า และระบุคลังให้กับทุกรายการสินค้า
  3. 3. ตรวจจำนวน สต็อก และข้อมูลจัดส่ง จากนั้นบันทึกเป็นร่างเพื่อทบทวน
  4. 4. ยืนยันเมื่อพร้อมดำเนินการ ระบบจะตัดสต็อกของรายการตามคลังที่ระบุ
ข้อควรระวัง: ใบสั่งขายแก้ไขหรือลบได้เฉพาะสถานะร่าง หากยกเลิกใบสั่งขายที่ยืนยันแล้ว หรืออยู่ระหว่างจัดส่ง/ส่งมอบ ระบบจะคืนสต็อกตามรายการเดิม
เอกสารต่อเนื่อง ส่งสินค้าและออกใบแจ้งหนี้ ▼

ใบส่งสินค้า

  1. สร้างจากใบสั่งขายที่ยืนยันแล้วหรืออยู่ระหว่างจัดส่ง/ส่งมอบ
  2. แก้ไขและลบได้เฉพาะร่าง
  3. ยืนยัน แล้วเปลี่ยนสถานะเป็นจัดส่ง และส่งมอบตามการทำงานจริง
  4. ยกเลิกได้ก่อนส่งมอบ

ใบแจ้งหนี้

  1. สร้างจากใบสั่งขายที่ยืนยันแล้ว หรือใบส่งสินค้าที่จัดส่ง/ส่งมอบแล้ว
  2. ตรวจงวดภาษี วันที่ ลูกค้า และยอดก่อนส่ง
  3. ส่งใบแจ้งหนี้เพื่อให้สถานะเป็น ส่งแล้ว
  4. ใช้เอกสารสถานะส่งแล้วสำหรับรับชำระเงิน

ระบบสร้างเอกสารต่อเนื่องที่ยังใช้งานอยู่ได้เพียงหนึ่งฉบับต่อใบสั่งขายหรือใบส่งสินค้า จึงควรค้นหาเอกสารเดิมก่อนสร้างใหม่

02 · คลังสินค้า

ควบคุมยอดตามคลังอย่างตรวจสอบได้

เมนูหลัก: คลังสินค้า › รับสินค้า, ปรับสต็อก, นับสต็อก, โอนสต็อก, คืนสต็อก, เบิกสต็อก, ยอดคงเหลือ, ความเคลื่อนไหวสต็อก

เอกสารสต็อก รับ เบิก คืน ปรับ และโอนสต็อก ▼

รับสินค้า

สร้างเอกสารรับเข้า เลือกคลังและสินค้า ตรวจจำนวน แล้วกดยืนยันเพื่อเพิ่มสต็อก สามารถสร้างร่างรับสินค้าจากใบสั่งซื้อที่ยืนยันแล้วหรือรับแล้วได้

เบิกและคืนสต็อก

ใช้เบิกสินค้าออกหรือบันทึกคืนสินค้าเข้าคลัง ตรวจคลังต้นทาง/ปลายทางและจำนวนก่อนยืนยัน ระบบตรวจยอดคงเหลือเมื่อตัดออก

ปรับสต็อก

ใช้แก้ยอดให้ตรงกับข้อเท็จจริง พร้อมเหตุผลที่ตรวจสอบย้อนหลังได้ ไม่ใช้แทนการรับหรือเบิกตามกระบวนการปกติ

โอนสต็อก

ระบุคลังต้นทางและปลายทางให้ถูกต้อง ยืนยันได้เมื่อคลังต้นทางมีจำนวนเพียงพอ

กติกาเดียวกัน: เอกสารรับ ปรับ โอน คืน และเบิก เริ่มจากร่าง เมื่อยืนยันแล้วจะบันทึกความเคลื่อนไหวสต็อก และเอกสารที่ยืนยันแล้วแก้ไขหรือลบไม่ได้
ตรวจนับและติดตาม นับสต็อก ยอดคงเหลือ และความเคลื่อนไหว ▼
  1. 1. เปิดเอกสารนับ: เลือกคลังและบันทึกยอดที่นับจริงในสถานะร่าง
  2. 2. ล็อกคลังเมื่อเริ่มนับ: หากเอกสารร่างล็อกคลัง ระบบจะไม่อนุญาตให้ยืนยันรายการเคลื่อนไหวสต็อกอื่นในคลังนั้นจนกว่าการนับจะเสร็จ
  3. 3. ยืนยันผลนับ: ตรวจจำนวนทั้งหมดก่อนยืนยัน ระบบปรับยอดตามผลนับและปลดล็อกคลัง
  4. 4. ตรวจย้อนหลัง: ใช้ยอดคงเหลือดูยอดปัจจุบัน และใช้ความเคลื่อนไหวสต็อกเพื่อตรวจเอกสารที่ทำให้ยอดเปลี่ยน

03 · จัดซื้อ

ขอซื้อ สั่งซื้อ และรับเข้า

เมนูหลัก: จัดซื้อ › ใบขอซื้อ, ใบสั่งซื้อ และ รับสินค้า

ขั้นตอนจัดซื้อ จากความต้องการจนของเข้าคลัง ▼

1. ใบขอซื้อ

สร้างร่าง ระบุสินค้า จำนวน และผู้ขายเมื่อทราบ ส่งอนุมัติเมื่อข้อมูลครบ ใบขอซื้อที่อนุมัติแล้วจึงสร้างใบสั่งซื้อได้

2. ใบสั่งซื้อ

ตรวจผู้ขาย ราคา จำนวน และเงื่อนไข ยืนยันตามลำดับอนุมัติขององค์กร แล้วติดตามสถานะจนรับสินค้า

3. รับสินค้า

สร้างร่างรับสินค้าจากใบสั่งซื้อที่ยืนยันแล้ว ตรวจของจริงกับเอกสาร และยืนยันเพื่อเพิ่มสต็อกในคลังที่เลือก

ใบขอซื้อและใบสั่งซื้อแก้ไขหรือลบได้เฉพาะร่าง ใบสั่งซื้อที่รับครบแล้วไม่สามารถยกเลิกได้

04 · การเรียกเก็บเงิน

ส่งใบแจ้งหนี้ รับชำระ และออกใบเสร็จ

เมนูหลัก: การเรียกเก็บเงิน › ใบแจ้งหนี้, รับชำระเงิน, ใบเสร็จรับเงิน

ลูกหนี้ ติดตามยอดจนปิดชำระ ▼
  1. 1. สร้างใบแจ้งหนี้เป็นร่าง แล้วตรวจงวดภาษี ลูกค้า และยอด
  2. 2. ส่งใบแจ้งหนี้ เพื่อให้เป็นสถานะส่งแล้วและพร้อมรับชำระ
  3. 3. สร้างรับชำระเงิน เลือกลูกค้าหนึ่งราย และเลือกใบแจ้งหนี้สถานะส่งแล้วที่ต้องการตัดยอด
  4. 4. ระบุยอดจัดสรรไม่เกินยอดค้างของแต่ละใบ แล้วกดยืนยันเพื่อบันทึกการรับชำระและปิดใบที่ชำระครบ
ใบแจ้งหนี้: แก้ไขหรือลบได้เฉพาะร่าง การส่งและยกเลิกมีผลต่อรายการบัญชีของเอกสาร
รับชำระเงิน: ยกเลิกรายการยืนยันแล้วเพื่อย้อนการจัดสรรและรายการบัญชีได้ แต่ลบได้เฉพาะร่าง

ใบเสร็จรับเงินเป็นเอกสารแยกสำหรับใบแจ้งหนี้ที่เชื่อมโยงอยู่ ให้ยืนยันหลังตรวจยอดรับจริงและเอกสารอ้างอิงครบถ้วน

05 · LINE

รับคำสั่งซื้อจาก LIFF และติดตามการแจ้งเตือน

สำหรับผู้ปฏิบัติงาน สิ่งที่เกิดขึ้นเมื่อมีคำสั่งซื้อผ่าน LINE ▼
  • LIFF จะแสดงเฉพาะสินค้าที่เปิดใช้งานและมีสต็อกในคลังเริ่มต้นขององค์กร
  • เมื่อชำระคำสั่งซื้อผ่าน LIFF สำเร็จ ระบบสร้างและยืนยันใบสั่งขายให้ จึงควรตรวจรายการจากเมนูใบสั่งขายตามกระบวนการขายปกติ
  • ระบบตรวจราคา จำนวน และสต็อกบนเซิร์ฟเวอร์ก่อนยืนยัน ไม่ใช้ราคาหรือคลังที่ส่งมาจากหน้าลูกค้าโดยตรง
  • หากลูกค้ามีการผูกบัญชี LINE แล้ว ระบบอาจส่งการแจ้งเตือนเมื่อสถานะใบสั่งขายเปลี่ยน หรือเมื่อยืนยันรับชำระเงิน
การส่งข้อความเป็นการแจ้งเตือนประกอบเท่านั้น หากส่งไม่สำเร็จ เอกสาร ERP ที่ยืนยันแล้วจะไม่ถูกย้อนกลับ ให้ตรวจสถานะเอกสารใน ERP เป็นแหล่งข้อมูลหลัก

Part 2 · ผู้ดูแลองค์กร

กำหนดสิทธิ์และการตั้งค่าก่อนใช้งานจริง

ภาคนี้สำหรับผู้ที่ดูแลการทำงานขององค์กรในระบบ ให้เปลี่ยนค่าทีละเรื่อง ตรวจผลกับผู้ใช้ที่เกี่ยวข้อง และบันทึกเหตุผลก่อนนำไปใช้กับเอกสารจริง การสร้างหรือแก้ไของค์กร บทบาทรวม สิทธิ์รวม โมดูล และค่า platform เป็นงานของผู้ดูแลแพลตฟอร์ม ไม่ใช่งานตั้งค่า ERP รายวัน

ก่อนเปลี่ยนค่า: ตรวจผลกระทบต่อผู้ใช้ เอกสารที่กำลังทำ และงวดที่ยังเปิดอยู่ก่อนเสมอ ห้ามเปลี่ยนค่าเพื่อแก้เอกสารที่ยืนยันหรือปิดงวดไปแล้ว

A1 · สิทธิ์

พนักงาน บทบาท และสิทธิ์ HRMS

เมนูหลัก: HRMS › พนักงาน, แผนก, บทบาท และ สิทธิ์ · เส้นทางองค์กรใช้ HRMS roles ผูกกับพนักงาน ไม่ใช่เมนู Admin ของแพลตฟอร์ม

กำหนดตามหน้าที่งาน ให้สิทธิ์เท่าที่จำเป็น แล้วตรวจผลจริง ▼
  1. 1. ตรวจข้อมูลพนักงาน แผนก และหน้าที่งานให้ตรงกับโครงสร้างที่ใช้งาน
  2. 2. สร้างหรือแก้บทบาท HRMS ให้มีเฉพาะ permission ของงานนั้น จากนั้นผูกบทบาทกับพนักงาน ไม่ใช้บทบาทสิทธิ์กว้างเพื่อแก้ปัญหาชั่วคราว
  3. 3. ตรวจเมนู รายการ และปุ่มสำคัญด้วยบัญชีที่ผูกพนักงานและบทบาทนั้น หรือให้ผู้ใช้เข้าสู่ระบบใหม่แล้วยืนยันผล
  4. 4. บันทึกว่าใครได้บทบาทใด และทบทวนทันทีเมื่อย้ายงาน หมดหน้าที่ หรือลาออก

สิทธิ์ระดับองค์กร

ผู้ใช้ทั่วไปได้ permission จากบทบาทแพลตฟอร์ม (ถ้ามี) รวมกับบทบาท HRMS ของพนักงานที่ผูกบัญชี เมนู HRMS ใช้สิทธิ์เช่น hrms.employees.list, hrms.employees.edit, hrms.roles.list, hrms.roles.manage

แยกจากผู้ดูแลแพลตฟอร์ม

เมนู Admin เช่น องค์กร บทบาทรวม สิทธิ์รวม และโมดูล เป็นงานแพลตฟอร์ม ผู้ใช้ระบบที่เป็น admin จะผ่านการตรวจสิทธิ์ทุกอย่าง อย่าใช้เส้นทางนี้แทนการจัดสิทธิ์พนักงานรายวัน

การบันทึกชุด permission ของบทบาท HRMS จะแทนที่ชุดเดิมของบทบาทนั้นทั้งชุด จึงต้องตรวจผลกระทบต่อพนักงานทุกคนที่ใช้บทบาทเดียวกันก่อนบันทึก หากไม่มีเมนู HRMS หรือจัดการบทบาทไม่ได้ ให้ขอ permission ที่ตรงหน้าที่จากผู้ดูแลแพลตฟอร์ม

A2 · ตั้งค่า ERP

ตั้งค่า ERP ข้อมูลหลัก และเลขลำดับ

เมนูหลัก: ตั้งค่า › ทั่วไป, สกุลเงิน, เลขลำดับ, ข้อมูลหลัก และ Audit Log

ค่าที่ใช้ประกอบเอกสาร จัดลำดับตรวจจากข้อมูลหลักไปยังเลขเอกสาร ▼

ตั้งค่าทั่วไปและสกุลเงิน

ตรวจรูปแบบวันที่/ปฏิทิน (ค.ศ. หรือ พ.ศ.) การแสดงสัญลักษณ์สกุลเงิน ค่าเริ่มต้นภาษี และการเปิดใช้หัก ณ ที่จ่ายก่อนเริ่มรอบงาน ค่าทั่วไปและการแสดงผลสกุลเงินผูกกับบัญชีผู้ใช้ จึงให้ผู้เกี่ยวข้องตรวจว่าบัญชีตนแสดงผลถูกต้อง รายการสกุลเงินและอัตราแลกเปลี่ยนอยู่ที่เมนูสกุลเงินแยกต่างหาก และสกุลเงินฐานลบไม่ได้

ข้อมูลหลัก

เพิ่มหรือแก้หมวด/ค่าข้อมูลหลักหลังตรวจผลต่อฟอร์มและรายงานแล้ว หมวดที่ระบบสงวนไว้ (System) ลบไม่ได้ อย่าลบหมวดหรือค่าที่เอกสารอ้างอิงอยู่เพื่อหลีกเลี่ยงข้อมูลอ้างอิงเสียหาย

เลขลำดับเอกสาร

กำหนดรูปแบบด้วย token วันที่และตัวเลข ใช้ตัวอย่างเลขถัดไปเพื่อตรวจก่อนบันทึก เลือกรอบรีเซ็ต (รายวัน รายเดือน รายปี หรือไม่รีเซ็ต) ให้สอดคล้องกับการใช้งาน เลขลำดับในตัวของระบบลบไม่ได้ และการเปลี่ยนรูปแบบหรือ reset มีผลกับเอกสารใหม่เท่านั้น เอกสารเดิมยังเก็บเลขเดิม

ตรวจย้อนหลัง

ใช้ Audit Log ตรวจการเปลี่ยนแปลงที่มีผลต่อการทำงานก่อนและหลังแก้ค่า เพื่อยืนยันผู้ดำเนินการ เวลา และค่าอ้างอิงที่เกี่ยวข้อง

ข้อควรระวัง: เลขลำดับและค่าทั่วไปถูกเก็บตามผู้ใช้ ไม่ใช่ลำดับหรือค่ากลางของทั้งองค์กร การแก้รูปแบบ รีเซ็ต หรือค่าแสดงผลจึงต้องประสานผู้ใช้ที่ออกเอกสารและตรวจเลข/ค่าถัดไปก่อนเสมอ

A3 · การควบคุม

ปีบัญชี งวดภาษี และเกณฑ์อนุมัติ

เมนูหลัก: บัญชี › ปีบัญชี, งวดภาษี และ ตั้งค่า › เกณฑ์การอนุมัติ

ปิดงวดและอนุมัติ ควบคุมวันที่เอกสารและวงเงินก่อนใช้งาน ▼
  1. 1. เปิดปีบัญชีและงวดภาษี: สร้างปีบัญชีและงวดภาษีให้ครอบคลุมวันที่ที่จะบันทึกรายการ ระบุชื่อและช่วงวันที่ให้ถูกต้อง และตรวจว่างวดที่ต้องใช้งานยังเปิดอยู่
  2. 2. ตรวจรายงาน VAT: ทบทวนเอกสารและรายการบัญชีที่เกี่ยวข้องก่อนสั่งปิดงวดภาษี
  3. 3. ปิดงวด: งวดภาษีที่ปิดแล้วไม่แก้ไขหรือลบได้ แต่เปิดกลับได้เมื่อจำเป็น ปีบัญชีที่ปิดแล้วไม่แก้ไขหรือลบได้ และไม่มีเส้นทางเปิดปีบัญชีกลับ วันที่ที่อยู่ในงวดภาษีหรือปีบัญชีที่ปิดจะถูกบล็อกไม่ให้บันทึกหรือแก้ไขเอกสารบัญชีที่เกี่ยวข้อง
  4. 4. ตั้งเกณฑ์อนุมัติ: ระบุประเภทเอกสาร ยอดขั้นต่ำ และ permission ผู้อนุมัติ (erp.purchasing.approve หรือ erp.bills.approve) แล้วทดสอบกับยอดต่ำกว่าและสูงกว่าเกณฑ์ก่อนแจ้งผู้ใช้งาน
ขอบเขตเกณฑ์: ใช้กับการยืนยันใบสั่งซื้อ และการเปลี่ยนสถานะใบรับวางบิลเป็นอนุมัติเท่านั้น ไม่ใช่ workflow อนุมัติหลายขั้นหรือคิวอนุมัติ
ผลบังคับใช้: ระบบเลือกเกณฑ์สูงสุดที่ไม่เกินยอดเอกสารในจังหวะดำเนินการ การแก้เกณฑ์ไม่ย้อนเปลี่ยนเอกสารที่ยืนยันหรืออนุมัติแล้ว

A4 · LINE

เชื่อมต่อ LINE และเปิดใช้ LIFF อย่างปลอดภัย

เมนูหลัก: ตั้งค่า › LINE Integration · ต้องมีสิทธิ์ erp.line-integration.manage

หนึ่ง connection ต่อองค์กร บันทึกค่า ตรวจคลัง และลงทะเบียน webhook ▼
  1. 1. เตรียม Messaging API channel ID, Bot user ID, LIFF ID, LINE Login channel ID, channel secret และ channel access token จาก LINE Developers Console
  2. 2. เลือกคลังเริ่มต้นที่เป็น active และอยู่ในองค์กรเดียวกัน เพราะระบบใช้คลังนี้แสดงยอดและตัดสต็อกของคำสั่งซื้อ LIFF
  3. 3. บันทึกค่าและเปิดใช้ LIFF ordering เมื่อต้องการรับคำสั่งซื้อ จากนั้นใช้ URL webhook ที่หน้าตั้งค่าแสดงไปลงทะเบียนใน LINE Developers Console
  4. 4. ตรวจว่า URL เป็น public HTTPS ที่ LINE เรียกถึง และตรวจคำสั่งซื้อผ่าน ERP หลังเปิดใช้จริง
ปกป้องข้อมูลลับ: ก่อนบันทึก credential เซิร์ฟเวอร์ต้องตั้ง LINE_CREDENTIAL_ENCRYPTION_KEY เป็นคีย์ base64 ความยาว 32 ไบต์ ห้ามวาง channel secret หรือ access token ในตั๋วงาน ภาพหน้าจอ หรือคู่มือนี้ เมื่อแก้ค่าภายหลังให้เว้นช่อง secret/token ว่างหากไม่ต้องการเปลี่ยนค่าเดิม
Configured ≠ ทดสอบ live: สถานะนี้ยืนยันเพียงว่ามีค่าครบและ credential ถูกเข้ารหัสเก็บ ไม่ได้เรียก LINE เพื่อตรวจ channel, webhook หรือ LIFF
ขอบเขต UI ปัจจุบัน: มีโหลดและบันทึก connection เท่านั้น ยังไม่มีปุ่มส่งข้อความทดสอบ ยกเลิก connection หรือล้าง/หมุน credential

ขอบเขตการขอสิทธิ์

หากไม่เห็นเมนูองค์กรหรือทำรายการ ERP/HRMS ไม่ได้ ให้ขอ permission ของฟังก์ชันนั้น (เช่น erp.settings.manage, hrms.roles.manage, erp.line-integration.manage) จากผู้ที่ดูแลสิทธิ์ในองค์กรหรือผู้ดูแลแพลตฟอร์ม อย่าพยายามเข้าผ่าน URL โดยตรง

งานระดับแพลตฟอร์ม เช่น สร้าง/แก้ไของค์กร บทบาทรวม สิทธิ์รวม โมดูลระบบ หรือผู้ใช้ admin ต้องส่งต่อผู้ดูแลแพลตฟอร์มเท่านั้น

คู่มือนี้อธิบายพฤติกรรมงานที่ระบบรองรับในปัจจุบัน โปรดใช้สถานะเอกสารและสิทธิ์ขององค์กรเป็นข้อมูลอ้างอิงเมื่อปฏิบัติงานจริง
